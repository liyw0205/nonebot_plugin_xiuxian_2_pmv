# 修仙插件后续开发路线

审计日期：2026-07-10

本文以当前 `main` 分支为准，替代早期以“计划实现”为主的架构审查。已经通过验收的
能力不再重复列为待办；这里只记录仍需维持的边界、必须处理的问题和建议顺序。

本文不得保存 Bot token、secret、用户 ID、群 ID或其他部署凭据。真实 Bot 参数只放在
不受 Git 跟踪的本地文档或环境配置中。

## 1. 当前基线

当前非 vendor Python 代码约 117,300 行、263 个文件；测试 138 项。已完成并有测试保护：

- `XiuxianPaths` 数据路径入口，以及业务源码相对 `data/xiuxian` 路径约束；
- startup/shutdown 生命周期、依赖和资源初始化退出普通 import；
- `UserRepository`、`EconomyRepository` 和旧数据管理器兼容外观；
- Web 模块显式依赖、启动期创建服务、统一 Bot 选择和消息投递；
- QQ group、C2C、channel、interaction、lifecycle 上下文；
- REFIDX、消息 ID、发送结果、审核、撤回和消息记录归一化；
- interaction ACK exactly-once、事件去重、有界任务队列和可靠性指标；
- Markdown、keyboard、按 AppID capability、模板有限热加载和文本降级；
- 媒体输入、缓存、SSRF/大小限制、重试和清理；
- JSON Store、HTTP Client、时间格式、Settings、TTL Store 等共享基础设施；
- vendored/installed/auto Adapter 选择、来源诊断和 QQ 双来源契约矩阵。

Adapter 基线和完整 QQ 验收证据见 `docs/architecture_acceptance.md`。直接发送的迁移边界见
`docs/message_delivery_migration.md`。

## 2. 必须处理：P0

以下问题具有明确的安全或数据正确性风险，应先于新的大型玩法和结构重构处理。

### P0-1 Web 管理面板身份认证

**进度：核心认证与默认安全配置已于 2026-07-10 完成；Web 终端独立授权并入 P0-3。**

**现状**

- 默认启用 Web 面板并监听 `0.0.0.0:5888`；
- 登录只要求提交一个位于 `superusers` 的管理员 ID；
- 登录后可使用数据库写入、消息发送、广播、撤回、备份恢复等高权限功能；
- CSRF、持久会话密钥、Host 白名单和功能开关已经存在，但不能替代身份认证。

**必须实现**

1. [x] 默认监听地址改为 `127.0.0.1`，外网监听必须显式配置；
2. [x] 登录增加独立密码哈希，不能只信任 ID；
3. [x] 未配置认证时禁止管理面板启动；
4. [x] 对登录失败增加限速和短期锁定，不记录密码、Cookie 或 CSRF token；
5. [x] 数据库写入、备份恢复和主动消息发送默认关闭；
6. [x] Web 终端增加二次确认或独立权限，不与普通面板登录等权；
7. [x] 增加 Flask test client 的密码、CSRF 和认证缺失测试；功能权限审计在 P0-3 完成。

**验收条件**

- 仅知道超级用户 ID 无法登录；
- 默认配置不能从非本机网络访问管理面板；
- 未认证请求不能触发任何写库、发送、恢复、更新或终端操作；
- README 包含反向代理、HTTPS、Cookie Secure 和密码配置示例，但不包含真实凭据。

### P0-2 `message.db` 路径与无损迁移

**进度：已于 2026-07-10 完成路径中心化、旧库备份迁移、冲突保护和回归测试。**

**现状**

`xiuxian_utils/message_db.py` 仍使用 `Path() / "message.db"`。其位置依赖进程 cwd；从仓库
根目录运行真实发送测试会在根目录产生 `message.db`、`message.db-wal` 和
`message.db-shm`，说明路径中心化尚未完全闭环。

**必须实现**

1. [x] 在 `XiuxianPaths` 增加 `message_db`，默认位于配置的数据目录；
2. [x] 消息记录、Web 查询、清理和容量统计只消费该路径；备份继续按策略排除临时消息库；
3. [x] 启动时检测旧 cwd 数据库并执行一次无损备份迁移；新旧同时存在时拒绝覆盖；
4. [x] 迁移使用文件锁、SQLite backup API、完整性校验、WAL/SHM 保留和失败回滚；
5. [x] 增加路径、旧库迁移、冲突和数据保留测试；
6. [x] 删除仓库根目录 `message.db*` 隐式依赖，并加入源码质量约束。

**验收条件**

- 任意 cwd 启动均使用同一个配置路径；
- 旧消息记录可见且计数一致；
- 迁移中断不会损坏或丢失旧库；
- 测试和真实 Bot 发送不会在源码目录生成数据库文件。

### P0-3 高权限 Web 操作的统一授权边界

**进度：已于 2026-07-10 完成全局端点权限注册、默认拒绝、终端短时二次认证和敏感操作审计。**

**现状**

路由大量重复检查 `"admin_id" in session`，权限语义散落；现有全局 feature gate 能关闭
部分能力，但登录、角色、功能开关和具体操作没有形成单一授权入口。

**必须实现**

1. [x] 建立统一 helper，区分只读、写库、消息、备份、更新、终端权限；
2. [x] 全局 `before_request` 成为认证和功能开关的安全边界，旧路由检查仅保留兼容；
3. [x] 对全部路由建立自动权限注册审计，写请求继续统一执行 CSRF；
4. [x] `/upload_image` 本机豁免只读取直接连接地址，不信任转发来源头；
5. [x] 对写库、消息、备份、更新和终端操作记录管理员、路由目标和结果，不记录秘密。

**验收条件**

- 新增高权限路由若未声明权限，测试直接失败；
- 权限拒绝统一返回 401/403，不通过异常或页面重定向掩盖 API 错误；
- 审计日志足以定位操作者和操作对象。

## 3. 必须持续收敛：P1

这些问题当前有兼容边界，不要求一次性重写，但必须建立可量化的递减目标。

### P1-1 完成消息投递迁移

**进度：普通回复、主动群聊、主动私聊和媒体响应已于 2026-07-10 完成迁移；剩余调用均为投递后端或明确的平台专用能力。**

当前仍有旧 `bot.send`、`call_api` 和平台专用发送，主要位于管理员、宠物、宗门、Boss、
娱乐媒体、布局提示、广播和 Adapter 兼容层。

要求：

- [x] 修正源码质量 allowlist，不再用宽泛的 `__init__.py` 或功能包前缀放行；
- [x] 迁移普通文本、Markdown、图片以及主动群聊/私聊发送；
- [x] 同步收紧测试 allowlist 和 `message_delivery_migration.md`；
- [x] 所有新业务发送继续强制经过 `MessageDeliveryService`；
- [x] 只允许 delivery backend、Adapter compatibility 和明确的平台专用 facade 直接调用。

### P1-2 统一运行时 HTTP 调用

**进度：事件处理和 Web 请求路径中的通用同步 HTTP 已于 2026-07-10 收敛到共享客户端。**

共享 `HttpClient` 已存在，但项目仍有直接 `requests`、`httpx.AsyncClient` 和
`aiohttp.ClientSession`。资源安装器或协议专用流式客户端可以保留，但运行在事件处理路径
中的同步网络调用必须迁移，避免阻塞事件循环。

优先审计：

- `xiuxian_admin/empty_fallback.py`；
- `xiuxian_entertainment/mod/anime_reaction.py`；
- `xiuxian_entertainment/mod/music_utils.py`；
- `xiuxian_info/draw_changelog.py`；
- `xiuxian_utils/external_api.py`；
- `xiuxian_web/core.py`、`xiuxian_web/messages.py`。

已完成：

- [x] 默认回复、动漫反应、点歌、更新日志、外部 ID/上传 API 和 Web 媒体代理使用共享客户端；
- [x] 事件循环中的同步调用均通过 `run_blocking_io`、`asyncio.to_thread` 或 Web 同步线程执行；
- [x] 增加重点运行时文件静态约束和共享客户端 timeout/retry/大小限制测试；
- [x] 保留媒体解析器的 `aiohttp` 会话，因为第三方解析核心要求共享该会话；
- [x] 保留媒体下载器、头像下载器的 `httpx`/`aiohttp` 流式客户端，用于 SSRF 校验、流式大小限制或二进制读取；
- [x] 保留更新、备份和 WebDAV 模块的 `requests`，这些是协议/流式工具并在 Web 线程或显式后台线程运行。

### P1-3 统一可变 JSON 状态

**进度：路线列出的高风险可变状态已于 2026-07-10 迁移到原子、带锁和损坏恢复的中央 Store。**

除中央 Store 外仍约有 27 个文件直接 `json.load/json.dump`。并非所有 JSON 都需要迁移：
只读配置和资源数据可保留直接解析；会被运行时修改的状态必须使用原子写、类型校验和锁。

优先迁移补偿、邀请、互动、幻境、活动配置、相缘、小游戏存档、宗门配置、工作奖励等
可变状态。迁移前为现有格式补 fixture，保证兼容旧文件和损坏恢复。

已完成补偿、邀请、互动、活动配置、相缘、扫雷、五子棋、十点半、宗门周购、工作奖励和
幻境状态；中央 Store 测试覆盖旧格式读取、根类型校验、损坏文件备份恢复、原子替换和并发
读改写。幻境额外覆盖旧用户存档补全、统计结构修复、并发计数，以及仅清理玩家存档时保留
统计文件。

### P1-4 关键经济流程行为测试

**进度：已建立原子灵石转账基线；2026-07-11 继续完成宗门成员管理，以及神物消耗与角色成长的事务、回滚和幂等测试。**

当前 138 项测试主要保护基础设施和关键仓储边界，对约 117,300 行玩法代码仍不构成全面
覆盖。继续拆分
宗门、交易、背包、突破或宠物前，必须先覆盖关键事务：

- 交易扣款、交付、取消、超时和重复提交；
- 宗门职位、贡献、踢出和权限；
- 背包增减、装备替换和失败回滚；
- 突破、奖励、签到、补偿的幂等性；
- Bot 重复事件和任务重试不得重复发放经济资产。

测试应使用临时数据库和明确事务，不依赖生产数据目录或真实 Bot。

首批测试使用临时 SQLite 验证扣款与入账在同一事务、收款方不存在时完整回滚、余额不足
不产生部分状态，以及同一操作 ID 重试不会重复发放资产。该能力通过兼容数据管理器暴露为
`transfer_ls`，后续交易、宗门和奖励流程可逐条迁移到这一事务边界。

宗门成员退出、管理者踢出和职位变更现已迁移到 `SectMembershipService`：权限与成员关系在
事务内按实时数据库状态复核，成员关系与贡献清零原子提交，职位人数上限在写入前校验，
并使用操作 ID 防止重复事件二次执行。行为测试覆盖越权、跨宗门、同级目标、职位上限、
重复提交和数据库异常回滚。

送灵石命令现已迁移到 `StoneGiftService`：发送方按赠送总额扣款、接收方按扣除 10% 手续费
后的净额入账，并在同一事务记录操作 ID。重复 Bot 事件返回首次结果且不重复扣款、入账或
累计每日额度；余额不足、接收方不存在及操作记录写入失败均保持双方资产不变。行为测试
覆盖成功、重复提交、失败回滚和非法参数。

鬼市存取灵石现已迁移到 `GuishiStoneService`：服务通过 SQLite 附加数据库将玩家灵石和
`trade.db` 鬼市余额纳入同一事务，取款手续费也按事务内实时余额计算。操作 ID 用于抵御
重复事件；任一数据库写入失败、玩家余额不足或鬼市余额不足时均不会留下跨库部分状态。

拍卖等待区上下架现已迁移到 `AuctionQueueService`：可交易背包物品扣减或绑定退回与
`trade.db` 等待区记录在跨库事务中同步完成，并在事务内复核排队上限、重复物品和实时库存。
重复消息不会二次扣物或退物，等待区或操作记录写入失败时背包与队列均完整回滚。

鬼市取物品现已迁移到 `TradeRepository.take_guishi_stored_item`：服务通过 SQLite 附加
`trade.db`，将鬼市暂存区删除、绑定物品入包和操作记录写入纳入同一事务。重复消息只复用首
次结果，不会重复入包；背包空间不足、暂存状态变化或操作记录写入失败时，暂存区和背包状
态都会完整回滚。

普通成长类神物和固定修为丹药使用已迁移到 `CultivationItemService`，物品扣减、修为与角色
属性增长、战力重算在同一事务内完成；同一消息重复投递只结算一次，物品不足或操作记录
写入失败不会产生部分角色成长。炼体结算时间类神物仍沿用其独立状态管理，不在本次事务
边界内混合处理。

三转玄丹已迁移到 `ThreeCultivationPillService`：随机修为结果在进入服务前确定，服务按事务内
实时修为执行境界上限截断，并原子完成道具扣减、修为与气血真元恢复、战力重算和操作记录；
重复消息沿用首次随机结果与结算数据，不会再次消耗玄丹。

突破概率丹药已迁移到 `BreakthroughRateItemService`，丹药扣减、耐药计数和突破率增长在同一
事务内提交；重复消息返回首次使用数量与增长值，物品不足或操作记录失败不改变突破率。

状态恢复类丹药已迁移到 `RecoveryItemService`，气血真元恢复、回满状态、体力恢复与背包扣减
在同一事务内提交，并保留耐药计数与绑定数量维护；重复消息不再次消耗丹药。

永久攻击丹药已迁移到 `PermanentAtkItemService`，`BuffInfo.atk_buff` 增长、丹药扣减与耐药计数
原子提交；重复消息沿用首次增长值，失败不会留下仅增长攻击或仅扣除丹药的部分状态。

解绑符已迁移到 `UnbindItemService`，服务在事务内按实时解绑符数量和目标绑定数量截断实际
解绑数，并原子完成解绑符扣减、目标绑定数更新和操作记录；重复消息不会重复解绑。

灵签宝箓已迁移到 `LotteryTalismanService`：随机奖励在事务外预先确定，服务原子完成宝箓扣减、
奖励入包和操作记录；重复消息复用首次抽取结果，不会再次消耗宝箓或重复发放奖励，结算失败
则完整回滚背包变化。

技能书确认学习已迁移到 `SkillLearningService`，技能书扣减、对应 `BuffInfo` 技能槽替换和操作
记录在同一事务内提交；重复确认不再次消耗技能书，写入失败不会留下只扣书或只换技能的
部分状态。

天命丹与天命渡劫丹融合已迁移到 `PillFusionService`：随机成功结果在事务外确定，服务原子
完成材料扣减、成功产物入包和操作记录；失败融合仍按规则消耗材料，重复事件复用首次结果，
不会再次消耗、发放或累计融合统计。

道号与灵根改名已迁移到 `PlayerRenameService`：灵石、易名符或灵根改名卡扣减与名称写入在
同一事务内完成，道号唯一性按事务内实时状态复核；重复事件不会重复扣费，名称冲突或写入
失败不会留下资产已扣但名称未更新的部分状态。

## 4. 结构治理：P2

P2 是维护性工作，不应压过 P0/P1，也不应一次性重写。

### P2-1 拆分超大模块

**进度：已于 2026-07-10 从 Web 核心提取独立访问控制模块，路由权限表与解析逻辑可单独测试。**

当前主要热点：

- `xiuxian_utils/xiuxian2_handle.py`：约 4,381 行；
- `xiuxian_sect/__init__.py`：约 3,004 行；
- `xiuxian_trade/__init__.py`：约 2,924 行；
- `xiuxian_web/messages.py`：约 2,286 行；
- 多个玩法 `__init__.py` 仍超过 1,000 行。

按“一条完整命令流”拆分 commands/service/repository/presenter，不按行数机械切文件。每次拆分
必须先有行为测试，保留兼容导入，并验证插件加载顺序和 matcher 注册数量不变。

本轮选择完整的 Web 授权行为边界而非机械切行：`access.py` 负责权限模型、端点注册和解析，
`core.py` 保留 Flask 请求编排与兼容导出。更大的交易、宗门和数据管理模块仍须在新增命令流
测试后逐条提取，不作为一次性重写目标。

### P2-2 扩展 Repository 边界

**进度：经济 Repository 已扩展原子、幂等转账边界，并由旧数据管理器兼容暴露。**

目前仍有约 134 个模块级 `XiuxianDateManage`、`PlayerDataManager`、`Items` 实例。不要一次
替换；沿已建立的 Repository 模式，从交易、宗门和背包中选择事务边界清晰的流程逐步迁移。

### P2-3 Web application factory

Web 已消除 `from .core import *`，但仍使用全局 Flask app 和导入注册路由。只有在开始拆分
Web、需要多实例测试或出现循环依赖时再引入 application factory/Blueprint；当前不是 P0。

### P2-4 异常处理和可观测性

**进度：Web 敏感操作、HTTP 重试、JSON 损坏恢复和经济事务已有结构化边界及回归测试。**

非 vendor 源码仍有大量宽泛 `except Exception`。优先处理经济事务、网络、备份恢复和
消息投递边界；区分可重试、用户输入、配置、数据损坏和程序错误。不要以追求计数为目的
机械替换。

## 5. 明确非目标

- 不引入第二套 Bot、Webhook、插件调度、Web 面板或通用 Hook 框架；
- 不把 ElainaBot 或其他完整框架 vendor 进项目；
- 不一次性删除 `XiuxianDateManage` 或重写全部玩法；
- 不为只读 JSON 资源强制套状态 Store；
- 不自行实现 QQ 分片上传，除非契约测试证明当前 Adapter 缺失必需能力；
- 不在仓库、测试 fixture、日志或提交历史中保存真实 Bot 凭据和用户/群标识。

## 6. 推荐执行顺序

1. **Web 认证默认安全化**：本机监听、密码哈希、高危能力默认关闭、认证测试；
2. **`message.db` 中心化与迁移**：路径、兼容迁移、跨 cwd 和 WAL 测试；
3. **Web 授权边界**：统一权限 helper、自动路由审计、敏感操作日志；
4. **发送迁移第一批**：管理员普通响应、宠物、宗门、Boss；
5. **运行时 HTTP 收敛**：先消除事件循环中的同步请求；
6. **可变 JSON 状态迁移**：按数据风险逐个迁移；
7. **关键经济流程测试**：为交易、宗门、背包、突破建立行为基线；
8. **按命令流拆分大模块**：有测试后再提取 service/repository/presenter。

每项仍执行：定向测试 -> 全量测试 -> `compileall` -> `git diff --check` -> 独立提交推送。
